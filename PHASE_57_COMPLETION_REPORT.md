# Phase 57 — AI Assistant (Foundation): Completion Report

## Scope

Full local, offline AI Assistant foundation per `AI_ASSISTANT_MASTER_PROMPT.md` and `PHASE_57_TECHNICAL_SPEC.md`: audit → real benchmark → packaging validation → spec sign-off → implementation → live verification, following this project's spec-first, approval-gated convention for phases touching money/customer/permission data. Executed as one continuous pass per explicit founder instruction ("keep going straight through") after spec sign-off, with real, load-bearing findings surfaced and fixed along the way rather than deferred.

## Real audit and benchmark (before any code was written)

A fresh, from-scratch re-audit (not trusting the 2026-07-03 original audit's numbers) found the codebase had grown substantially since this initiative was first scoped — Hotel/Lodge, Jewellery, Rental, Diagnostic Lab, and Blood Bank verticals shipped, Cash Book/Trial Balance/P&L reports added, international currency/tax generalized, a UPI-QR country-matching bug fixed. Of the original 19 pinned query templates, 6 had no backing function at all and 3 more needed extension — a materially different picture than "mostly exists, minor drift."

**The performance benchmark was run for real, not estimated**, on representative hardware (HP Pavilion laptop, i5-1155G7, 4c/8t, 15.8GB RAM, standard Windows Defender active) — deliberately closer to the low end of what a real small-shop owner would have than a dev workstation. This surfaced two real, load-bearing problems before any product code was written:

1. **The original model pin (Phi-3.5-mini-instruct) failed the real benchmark.** 0.05–0.6 tokens/sec, 5.1GB peak RAM — a single answer took 40–70 seconds and would have left too little headroom on an 8GB machine, which the founder explicitly required this to support. GPU acceleration, tried as a fix, crashed outright (`ErrorOutOfDeviceMemory`) on this machine's integrated GPU — confirming `gpu: false` must be explicit in code, never left to library auto-detection.
2. **A real bug in the benchmark itself**: `createContext()` with no explicit size defaults to near the model's full 131,072-token trained context, causing a 9.4GB RAM spike unrelated to the model choice. Fixed by explicitly bounding context to 4096 tokens — carried into the shipped code, not just the benchmark script.

**Model re-pinned to Qwen2.5-1.5B-Instruct** (Apache 2.0, re-verified) after it measured 1.8GB peak RAM (vs. 5.1GB) and ~2.2 tokens/sec once generating (vs. ~0.5) on the same hardware — a real, multi-x improvement on both axes that matter for the founder's 8GB requirement, not a marginal one.

## Packaging — validated end-to-end before touching the real project

Built an isolated Electron 33.3.1 + electron-builder 26.15.3 test app matching Sarang's exact config shape, and actually ran the packaged `.exe` (not just inspected the file layout). Confirmed: a plain `npm install node-llama-cpp` pulls in 695MB of unneeded platform/GPU variants (pruned to the 45MB CPU-only build); the existing generic `asarUnpack: ['**/*.node']` glob picks up the native binary automatically, no config change needed; `node-llama-cpp` resolves its own native binary internally, unlike Prisma's query-engine DLL which needs manual `process.resourcesPath` pointing in `db.ts`; real additive installer cost is 1.03GB (941MB model + 45MB binary + 42MB JS bundle), matching the arithmetic estimate. Also found `node-llama-cpp` v3 is ESM-only while Sarang's main process compiles to CJS — confirmed a dynamic `import()` bridges this cleanly, and that this isn't a new pattern (matches an existing lazy-import convention in `product.service.ts`).

## Implementation

- **`AIProvider` interface** (`ai-provider.ts`) with a `FakeAIProvider` test-double, proving the pipeline never depends on `node-llama-cpp`'s concrete API.
- **`NodeLlamaProvider`** (`ai-llama-provider.ts`) — grammar-constrained JSON output for intent classification (`LlamaJsonSchemaGrammar`), so the model cannot produce anything outside a fixed template enum, never free text, never SQL.
- **A genuinely separate, strictly read-only DB connection** (`ai-readonly-db.ts`), `mode=ro` + `PRAGMA query_only=ON`, distinct from the app's normal read-write client.
- **The two-call pipeline** (`ai-query.service.ts`): intent classify → template execution → business-rule validation → NL phrasing. 18 static templates (Sales/Inventory/Customers/Suppliers/Credit/Finance) plus dynamic active-vertical templates (`ai-vertical-templates.service.ts` — Hotel occupancy, Jewellery stock/sales, Rental status/revenue, Lab throughput, Blood stock), wired only for the one business type actually installed.
- **5 new aggregation functions** (`ai-aggregations.service.ts`: dead stock, bottom-revenue products, top customers by revenue, customers with no recent purchases, top suppliers by purchase volume) plus 1 extended existing function (`getTopProducts` gained optional date-range params), following `report.service.ts`'s established "call `getPrisma()`, aggregate in JS" style, including its RETURN-invoice sign-correction convention.
- **New `AiQueryLog` table** (additive-only, migrated safely onto the real dev DB — backed up first, applied via the established hand-written-migration pattern, zero data loss confirmed: 11 customers intact before and after).
- **IPC surface** (`ai.handler.ts`): `ai:query`, `ai:getStatus`, `ai:clearHistory` — every handler checks `isModuleEnabled('ai_assistant')` server-side **in addition to** `requirePermission()`, a genuinely new pattern with zero existing precedent (confirmed via a fresh grep of all 109 other handler files finding zero exceptions to the UI-only convention elsewhere).
- **New `ai.query` permission**, deliberately Admin/Manager only — the query catalog spans profit, supplier purchase volume, and full customer/credit visibility in one surface, which Cashier/Staff's normal screens don't show in aggregate; granting it more broadly would be exactly the side-channel the spec's permission-parity requirement forbids.
- **Settings UI, sidebar entry, chat panel** — `AiAssistantSection` (mirrors `BarcodeSection`'s toggle shape), "Ask Sarang" sidebar item (off by default, zero footprint when disabled), `AiAssistantScreen` chat panel with an explicit "Thinking..." state (cold-start latency is real, ~30–65s measured live — the UI doesn't pretend otherwise).

## Two real bugs found by live verification, not the original design or unit tests

**1. Real vertical-template grammar gap.** While wiring active-vertical templates, realized the model's grammar enum was static and never included vertical template names (`hotel.occupancy` etc.) — meaning the model could never have produced them in the first place, regardless of which business type was active. Fixed by changing `classifyIntent`'s signature to accept the caller-computed valid template list (static catalog + whichever vertical templates apply to the actual installed business type) and building the grammar from exactly that list per call. A new test confirms a hotel-only template is correctly refused for a non-hotel business, and correctly answered when the business type genuinely matches.

**2. Real model refusal failure, caught only by live testing against the actual model.** The FakeAIProvider unit tests all passed — but asking the real, packaged app "Should I file a GST return this month and what tax rate should I use?" got the real model to misclassify it as `sales.compareToPreviousPeriod` and answer with real sales figures instead of refusing, because the question also contained data-adjacent words ("this month"). This is exactly the failure mode Section 4 of the master prompt already anticipated in principle ("the model is never asked to decide not to answer in free text — that's promptable/injectable and not a real guarantee") and Section 7 named the sanctioned remedy for ("a fallback rule-based/keyword classifier for common phrasings") if the model underperforms — now proven necessary by a real failure, not built speculatively. Fixed with a deterministic, code-owned keyword safety net (`isDeterministicallyOutOfScope`) that runs **before** the model is even invoked, deliberately biased toward over-refusing per the spec's own stated tolerance. Re-verified live against the real model: now refuses correctly, and does so near-instantly since the model never gets called for an obviously out-of-scope question. Two regression tests lock in the exact real failure case and confirm the filter doesn't over-trigger on a legitimate question sharing similar vocabulary.

## Deliberately reduced scope, disclosed not hidden

The master prompt's golden-question eval harness calls for ~85 in-scope + ~15 adversarial questions run against the real classifier. Given real, measured per-question latency (30–65 seconds against the actual model), the full 100-question battery would require on the order of 1–1.5 hours of pure inference time. Given that constraint, this phase substituted **4 targeted live scenarios against the real running app** (fallback, real-answer, refusal, sidebar/settings gating) rather than the full battery — chosen to cover the highest-risk paths (hallucination prevention, real-data correctness, the safety/refusal layer, and the opt-in gating contract) rather than breadth across all 18+ templates. This is a real, acknowledged scope reduction, not a silently skipped requirement — and it's what caught the one real safety bug (finding #2 above), so the reduced scope still did its job. A fuller golden-eval pass remains a legitimate follow-up before treating template-classification accuracy as fully proven across the entire catalog, particularly the less-tested Inventory/Supplier templates and the 5 vertical templates beyond Hotel.

## Addendum (2026-07-13, same day) — response-time optimization pass

Founder asked to reduce response time, with a real measurement first. Three changes were tested for real, not assumed:

1. **Shortening the classification prompt text** (removing the full enum listing, relying on the grammar alone) — negligible speed change (37.3s vs 37.1s isolated) **and caused a genuine misclassification** (a legitimate "who owes me money" question was wrongly refused). Discarded, never shipped — a real example of a plausible-sounding optimization that would have quietly traded correctness for no actual benefit.
2. **Simplifying the classification grammar to just the `template` field**, dropping `dateFrom`/`dateTo`/`topN`/`days` — the model generated all four every time regardless of relevance (including fabricated placeholder dates for questions that never asked for a date range), which was pure wasted generation time on a CPU-bound bottleneck. Every template's own `execute()` function already has sensible built-in defaults, so no capability was lost. **Real, repeatable ~16% speedup on the classify call (25.1s → 21.2s isolated), zero accuracy cost** across multiple retests. Shipped.
3. Confirmed live in the real packaged app: `credit.whoOwesMe` question, correctness fully preserved, **48.2s live end-to-end** (down from two earlier live measurements of 52.7s and 64.8s).

**Root cause identified, not just patched**: the dominant cost is raw CPU-bound token-generation throughput (~2.2 tokens/sec on the benchmarked hardware), which is largely fixed regardless of prompt or schema trimming — the ~16% win came specifically from generating fewer output tokens, not from any prefill/prompt-length reduction (which the failed experiment #1 disproved). Getting meaningfully faster than this would require a smaller model or more aggressive quantization, both of which need their own full re-validation (the same rigor that already caught one real refusal-accuracy failure on the current model) — not attempted in this pass, named as the next lever if the founder wants to pursue it.

Test suite re-confirmed clean after the change: 0 TypeScript errors, 1133/1133 unit tests, 0 lint errors.

## Addendum 2 (2026-07-13, same day) — structural latency fix + business-type coverage expansion

Founder set a hard target: ≤5-10s response time, and asked whether the assistant supports all business types across Sarang. Two more real, structural changes:

**1. Removed the second (phrasing) LLM call entirely.** Every template already builds a natural-reading `headline` + `details` — the phrasing call was only ever restating them, never allowed to change the numbers (Section 3's own constraint). Replaced with `formatDeterministicAnswer()`, a plain string join — zero risk of the model altering a figure, one fewer model call per question, unconditionally.

**2. Added a deterministic keyword fast-path** (`tryFastPathClassify`) that skips model classification entirely for common, unambiguous phrasings, falling back to the LLM only when nothing matches. Conservative by design — specific patterns, not broad guessing, since a wrong fast-path match has no LLM judgment downstream to catch it.

**Real, live-measured result**: fast-path questions now answer in **~1 second** (from ~48-65s) — a ~50x improvement for the common case. LLM-fallback questions (deliberately unusual phrasings) now take **~27 seconds** (down from ~48-65s, roughly half, since the phrasing call is gone) — still over the 10s target. Honest limit reached: this is close to the floor of what CPU-only inference at ~2.2 tokens/sec can do for a genuine model classification call; going lower would need a smaller model (its own full re-validation) or hardware/architecture changes outside this pass's scope.

**Real formatting bug found by this change, not before it**: removing the LLM phrasing exposed that `print.service.ts`'s `formatAmount()` has no thousands separator ("₹18450.00", not "₹18,450.00") — previously masked because the LLM tended to reformat numbers naturally when generating text. Fixed with a local `formatAmountForSpeech()` wrapper (`ai-format.util.ts`), not by touching the shared `formatAmount()` (used across real invoice/receipt printing — different, wider-blast-radius change, out of scope here).

**Business-type coverage expanded from 5 to ~30 of 43 business types.** The 18 universal templates already worked for all 43 (shared Invoice/Product/Customer/Supplier model); vertical-specific questions only existed for Hotel/Jewellery/Rental/Diagnostic Lab/Blood Bank. Wired in 14 more already-existing, already-tested report functions (`ai-vertical-templates.service.ts`): Restaurant (food cost, order volume), Manufacturing (production), Electronics (serial/warranty), Clothing/Footwear (variant stock), Coaching Institute (test scores), CA Firm/Company Secretary (compliance tasks), Repair/Car Service (job cards), Pharmacy/Agri Inputs (batch expiry), 7 project-based verticals (Service/Consultant/Independent Consultant/Architect/Civil Engineer/Marketing Agency/Real Estate), 13 appointment-based verticals (clinics, salon, gym, driving school, lawyer, photo studio, event management, tailor, pest control — appointment utilisation + client retention, plus commission for Beauty Salon/Gym Studio), and Distributor/Hardware (logistics summary). Genuinely still uncovered: Retail, General, Placement Agency — no dedicated report function exists yet, disclosed rather than hidden.

**A second real classification bug found during this expansion, not before shipping it**: live-testing "How is production going this month?" on the dev DB's Manufacturing business type got the model to misclassify it as `sales.compareToPreviousPeriod` instead of the new `manufacturing.production` template — sales templates dominate the catalog numerically and "this month" pattern-matches strongly on them. This confirms newly-added vertical templates cannot be assumed correctly classified just because they compile and their backing functions work; live classification testing is required per template, not just per catalog. Fixed by extending the fast-path pattern list to cover all 14 new vertical templates (same deterministic-first-model-fallback shape as the original fix) — re-verified live, now correctly routes and does so near-instantly (26ms, fast-path, no model call). A regression test locks in this exact case.

Final state after both additions: 0 TypeScript errors, 1136/1136 unit tests, 0 lint errors, all live scenarios re-verified against the real running app.

## Addendum 3 (2026-07-13, same day) — feature restoration, home-screen placement, one more vertical, and an honest scope answer

Founder's follow-up, verbatim: no compromise on accuracy, no feature cut, the assistant should be able to answer "anything" related to a business, coverage should span every business type, and it should be reachable from the home screen. Addressed each directly rather than just reassuring:

**1. Restored the parameter-extraction capability Addendum 2 removed.** The grammar-simplification fix (Addendum 2, `ai-llama-provider.ts`) dropped `dateFrom`/`dateTo`/`topN`/`days` from the model's classification output for a real, measured speed win — that was a genuine capability reduction, not a no-op, and every template reading those params had silently been falling back to its hardcoded default ever since. Fixed with a new deterministic `extractParams()` (`ai-query.service.ts`) — regex for `topN`/`days`, real date arithmetic for `today`/`yesterday`/`this week`/`last week`/`this month`/`last month` — run on every question after classification regardless of which path (fast-path or LLM) matched. Deterministic extraction is strictly more reliable here than asking a 1.5B model to compute date ranges, and costs microseconds instead of reintroducing the model-generation time that caused the original slowdown. Three new regression tests assert the exact extracted values reach `getTopSuppliersByPurchaseVolume`, `getDeadStock`, and `generateProfitAndLossReport`'s `dateFrom`/`dateTo` — not just that extraction runs, that the number/date actually lands on the downstream call.

**2. Added a Dashboard entry point.** "Ask Sarang" now appears as a full-width, first-thing-visible banner directly under the Dashboard header (`DashboardScreen.tsx`), not just reachable via the sidebar — gated on the same `ai_assistant` module + `ai.query` permission check as everywhere else. Live-verified: banner renders, click navigates to `#/ai-assistant`.

**3. Closed one more of the three remaining vertical gaps.** Placement Agency had no vertical template despite `placement.service.ts`'s `getPlacementKPIs()` already existing and fitting the exact same reuse pattern as every other template — wired as `placement.summary`. Retail and General remain genuinely uncovered by a vertical-specific template, but this was re-examined, not just re-asserted: both have no dedicated data model beyond the shared Invoice/Product/Customer/Supplier tables every business type already uses, so there is no vertical-specific question left to wire — the 18 universal templates already answer everything a generic retail/general question would ask. That's the accurate state, not a gap to close.

**4. Honest answer on "the assistant should answer anything."** It cannot, by design, and that design choice shouldn't be walked back: the two-call architecture deliberately never lets the model write or execute a query (Section 3/4) — every answerable question must correspond to one of a fixed set of templates backed by a real, tested function. This is what makes "the model can never fabricate a number" a guarantee rather than a hope. The honest scope is: any question that maps to the 18 universal templates or the business's active vertical template(s) gets answered correctly from real data; anything else (a question needing a query the catalog doesn't have, or genuine advice) gets the fixed refusal/fallback message, never a guessed answer. Expanding the *set of answerable questions* is real, incremental, reviewable work (exactly what Addendum 2 and this addendum did) — not something a prompt instruction can make unlimited without removing the safety property that makes the assistant trustworthy in the first place.

Final state after this addendum: 0 TypeScript errors, 1140/1140 unit tests, all three live scenarios (dashboard banner + navigation, days-extraction fast-path, topN-extraction fast-path) re-verified against the real running app.

**Follow-up, same day**: the Dashboard entry point was upgraded from a plain navigate-only banner to a real question input — the owner types the question directly on the Dashboard (`aiQuestion` state + `handleAskFromDashboard` in `DashboardScreen.tsx`), which hands it to the dedicated AI Assistant screen via router state (`navigate('/ai-assistant', { state: { initialQuestion } })`). `AiAssistantScreen.tsx` auto-asks it on arrival via an effect keyed on `location.key` (React Router's per-navigation-entry id), so it fires exactly once per hand-off and never re-fires on a later plain sidebar visit (which carries no state). ## Addendum 5 (2026-07-13, same day) — the 3-way outstanding-balance discrepancy, investigated to root cause

Founder asked for the discrepancy flagged in Addendum 4 (₹611,984 vs ₹29,550 vs ₹6,850 for "customer outstanding balance") to be properly investigated, and for the "Thinking..." wait message to be corrected — see below for both.

**Investigation.** Traced every production code path that writes `Customer.outstandingBalance` (`billing.service.ts`, `payment.service.ts`, `credit-note.service.ts`, `quotation.service.ts` — all via the shared `customerLedgerService.addEntry()`, which atomically writes a `CustomerLedger` row and the mirrored denormalized column together; `returns.service.ts` and `import.service.ts`'s opening-balance import replicate the same atomic pattern by hand). **All of these are correct and mutually consistent by construction** — no production-code bug was found causing the column to drift from the ledger in normal app usage.

The actual divergence traced entirely to three orphaned test-fixture customers left in the dev database:
- Two customers (`UAT Test Client` ₹17,700, `UAT Rental Test Customer` ₹5,000) were soft-deleted (`isActive = 0`) by `tests/e2e/harness.js`'s `cleanupByNamePrefix()` helper, whose fallback path — reached when a hard `DELETE` fails on an FK constraint — ran `UPDATE Customer SET isActive = 0` but never reset `outstandingBalance` back to 0, even though the same function deletes the customer's `CustomerLedger` rows immediately before that. A soft-deleted customer with a stale nonzero balance and zero backing ledger history. **Fixed**: the fallback now also sets `outstandingBalance = 0`.
- A third customer (`undefined Heavy Ledger Customer`, ₹582,434 — ~95% of the total discrepancy) was a leftover fixture from `tests/e2e/suites/09-stress.js`'s bulk 5,000-row `CustomerLedger` stress test. **Result, verified**: all three sources now agree exactly — ₹6,850 across 2 customers, whether computed from the `Customer.outstandingBalance` column sum, the live `CustomerLedger` debit/credit aggregate (the Dashboard KPI card's method), or `Invoice.balanceAmount` (the AI/Reports method). Live-verified: Dashboard KPI card now shows "Outstanding: ₹6.8K" and the AI answers "₹6,850.00" for the identical question, matching to the cent.

**"Thinking..." message fix.** The copy said "this can take up to a minute **on the first question**" — inaccurate per Addendum 4's own measured data: the ~20-30s cost is the real model-inference call itself, not a one-time load; it's exactly as slow on the 10th non-fast-path question as the 1st (only the model's initial load into memory is genuinely first-question-only, and that's not even the dominant cost). ## Addendum 6 (2026-07-15) — navigation help, data-driven suggestions, and a capabilities disclosure

Founder asked for three things: (1) if a user is struggling to navigate the app, the AI should help; (2) it should be able to give suggestions using data when asked; (3) it should be able to state what it can and can't do when asked. A fourth ask — answering in all 13 languages — was explicitly declined by the founder after being shown the real cost/risk (the model is already ~2.2 tokens/sec on 8GB RAM hardware, ~48s per real query, and Qwen2.5's own documentation doesn't strongly claim the 6 Indian regional languages Sarang is proudest of supporting) — English-only stays as-is, unchanged from this phase's original decision.

**1. Capabilities disclosure — `meta.capabilities`.** A new deterministic fast-path template (never calls the model, same "fixed, code-owned text" principle as `REFUSAL_MESSAGE`/`FALLBACK_MESSAGE`) that states the real, current scope: the business-data categories it answers, the explicit refusal categories, that it can point to Manual chapters for navigation help, and the English-only limitation. Triggered by phrasings like "what can you do", "what can't you do", "help".

**2. Data-driven suggestions — `meta.suggestions`.** Restates `analytics.service.ts`'s existing `getDashboardAlerts()` output (the same alerts the Dashboard's own alert tiles show — low stock, overdue backup, large outstanding, pending reminders, audit-log failure, overdue rentals) as a natural-language answer. Zero new business logic, zero risk of the model inventing a numeric suggestion it can't justify — deliberately narrower than open-ended "give me advice," which a 1.5B local model isn't reliable enough for. Handles the zero-alerts case with its own "nothing needs your attention" headline rather than the generic `FALLBACK_MESSAGE` (which would misleadingly read as "couldn't find the data" rather than "genuinely all clear").

**Real design conflict found and resolved**: the existing `isDeterministicallyOutOfScope` safety net's `/\bshould i\b/i` pattern (added for a real live misclassification bug, deliberately biased toward over-refusing) runs *before* the fast-path and would have refused "what should I pay attention to" as advice-seeking. Rather than loosen a safety net that exists because of a real prior bug, the suggestions trigger phrasings were written to avoid the word "should" entirely ("what needs my attention", "any suggestions", "things to review") — a test locks in that this phrasing actually reaches `meta.suggestions`, not the refusal path.

**3. Navigation help — answered entirely client-side, never sent through the model.** The model has no knowledge of the app's UI (its templates only cover business-data questions), so asking it "how do I create an invoice" would either refuse or hallucinate. Instead, `AiAssistantScreen.tsx` intercepts questions matching a "how do I / where is / how to" pattern *before* calling the IPC at all, and searches the Phase 56 Manual's own chapter titles and body content (deterministic keyword matching, term-frequency scored — never generated text) for the best match, replying instantly with a link to the real, already-written Manual chapter. A real live bug was found and fixed during verification: initial title-only matching missed "how do I create an invoice" entirely (the word "invoice" isn't in the title "Billing & Documents"), so body content was added to the search; a second bug was found where distinct-word-presence scoring picked `getting-started.md` over `billing.md` (both mention "invoice" at least once, and the tie went to whichever chapter came first in the manifest array) — fixed by scoring on term *frequency* instead, so the chapter that's actually about the topic wins over one that mentions it in passing.

**Verification**: 3 new unit tests (capabilities fast-path, suggestions with real alert data, suggestions on the zero-alerts case) — 1147/1147 total, 0 regressions. 0 TypeScript errors both configs. Live-verified end-to-end in the real Electron app with real dev-DB data: capabilities answer matches design, suggestions answer correctly reflects real seeded alerts (2 low-stock products, a 12-day-overdue backup, 45 pending reminders), and the navigation match correctly routes "How do I create an invoice?" to the Billing & Documents chapter with a working "Open in Manual" link, confirmed landing on the right chapter.

Final state: 0 TypeScript errors, 1147/1147 unit tests, all three new capabilities live-verified.

## Addendum 7 (2026-07-15) — template-catalog expansion: 70 new templates, plus a real pre-existing timezone bug found and fixed

Founder asked for the AI Assistant to answer "ALL the questions which are related to his business and for the data we have." Rather than a literal enumerated question list (mostly paraphrases of the same underlying data), this addendum is a real audit of every existing template (40: 18 universal + 20 vertical + 2 meta from Addendum 6) cross-referenced against every `report.service.ts` function and every business type's actual tracked data model, to find genuine capability gaps — then closing them.

**Honest scope caveat, stated up front and unchanged from the original design**: no fixed template catalog can answer a truly arbitrary multi-dimensional question ("sales of product X to customers in city Y last quarter excluding returns") — that needs the model writing its own queries, which this architecture deliberately never allows (Section 3/4's "the model never writes or executes SQL" guarantee). This addendum covers every *realistic* single-entity or single-metric question across every business type; anything genuinely outside that still correctly falls to the existing refusal/fallback, not a guess.

**70 new templates added, in four batches:**

**Foundational fix (unlocks every date-scoped question at once).** `extractParams()` (`ai-query.service.ts`) previously only recognized `yesterday/today/this week/last week/this month/last month`. Added `tryParseSpecificDate()` — a specific calendar date ("15th Jan", "Jan 15 2025", ISO `YYYY-MM-DD`, day-first slash `DD/MM/YYYY`; a year-less date rolls back to the most recent past occurrence if it would otherwise land in the future, since a backward-looking sales question about a bare "15th Jan" almost certainly means the year that already happened) — plus `this year`/`last year` branches. Also fixed `getTopProducts()` (`analytics.service.ts`) so its `sortBy: 'quantity' | 'revenue'` parameter actually branches the sort (it was accepted but silently ignored, always sorting by revenue).

**Tier 1 — 12 templates, trivial reuse of an existing report function into a new template, zero new aggregation logic:** stock value, out-of-stock count, cash-in-hand balance, expense breakdown by category, biggest expense category, tax collected this period, staff attendance today, staff on leave, pending quotations count, pending purchase orders, credit/debit notes issued this period, total customer count.

**Tier 2 — 5 templates, a new capability shape (look up ONE record by identifier):** invoice by number, purchase order by number, customer by name/phone, supplier by name, product by name/SKU. A shared `extractSearchTerm()` helper feeds all five: tries a quoted string first (most unambiguous), then the longest digit-bearing code-like token (covers invoice/PO numbers, SKUs, phone numbers), then falls back to a run of Title-Case words (a bare name) — returns `null` rather than guessing if nothing is found, so the template's own empty-result path produces the standard fallback message instead of a wrong match.

**35 universal-deepening templates**, one level deeper than the original 18, spanning sales (8: category/hour-of-day/unique-customers/discounts/returns/cancelled/walk-in-split/quantity-based best-seller), inventory (7: value-by-category/near-reorder/SKU-count/added-this-month/never-purchased/turnover-rate/biggest-adjustment), customers (5: highest-purchase/average-spend/by-city/repeat-rate/new-this-week), suppliers (3: inactive/lead-time/purchase-value-this-month), finance (6: net-GST-payable/cash-vs-bank/discount-impact/net-worth-snapshot/expense-trend/profit-trend), staff (3: salary-paid/best-worst-attendance/active-headcount), and documents (3: open-quotations-value/conversion-rate/overdue-POs). **18 vertical-specific templates**, one per business type from a fresh gap audit (Lawyer, CA/CS, Architect, Civil Engineer, Real Estate, Marketing Agency, Software Agency, Photo Studio, Event Management, Driving School, Tailor/Boutique, Pest Control, Vet Clinic, Dental Clinic, Car Service Center, Coaching Institute, Diagnostic Lab, Placement Agency) — every one already had a dedicated, previously-unwired service file or KPI function (legal cases/hearings, drawing revisions, site visits, property KPIs, issues, shoot bookings, event bookings, driving tests/package balances, tailoring due-dates, pest contract renewals, vaccination due-dates, dental recalls, car job cards, coaching fees/attendance, ROC filings, candidate status). One template — unbilled time value — is shared across Lawyer and every project-based vertical (`TimeEntry.caseId`/`TimeEntry.projectId` are mutually exclusive per row, so one query correctly covers both without double-counting). Two more honest gaps: Marketing Agency gets no new template (no ad-spend/campaign-tracking model exists — "active campaigns" is honestly just its existing generic project list); Placement's "pipeline by stage" is reinterpreted as a `Candidate.status` breakdown (no dedicated Submission/Interview-stage model exists).

**Deliberate scope decision**: none of these 70 templates got a `FAST_PATH_PATTERNS` regex entry. Fast-path is a pure latency optimization for the clearest, most commonly-asked phrasings (per its own design) — every new template is still fully reachable via LLM classification (its name is automatically included in `availableTemplates`). Adding 70 more regexes would also multiply collision risk against the existing out-of-scope safety net and against each other.

### Real finding: a pre-existing timezone bug, verified and fixed

While writing a regression test for the new specific-date parser, discovered that `Date.prototype.toISOString().slice(0, 10)` — used throughout `ai-query.service.ts` and `ai-vertical-templates.service.ts` to stringify a **local calendar date** — silently shifts the date backward by one day on any positive-UTC-offset machine, because `toISOString()` converts to UTC. **Reproduced directly on the dev machine** (Asia/Calcutta, IST = UTC+5:30): `new Date(2025, 0, 15).toISOString()` returns `"2025-01-14T18:30:00.000Z"`, so `.slice(0, 10)` silently returns "2025-01-14" for a question about January 15th. This is this app's primary market (India/IST), not an edge case.

**Blast radius**: every `this month`/`last month`/`this week`/`last week`/`today` date-range default in `ai-query.service.ts`, plus `ai-vertical-templates.service.ts`'s pre-existing `thisMonthRange()` helper — used by most of the *original* Phase 57 vertical templates (jewellery, rental, restaurant, manufacturing, coaching, compliance, repair, service.projects, and more), not just this addendum's new ones. All of them were silently computing `dateFrom` one calendar day too early for any IST business, always — this wasn't limited to a midnight edge case for the month/week boundaries (only "today"/"now" stringification was specifically midnight-window-dependent).

**Fix**: added a `toLocalISODate(d: Date): string` helper — builds `YYYY-MM-DD` from `getFullYear()`/`getMonth()`/`getDate()`, zero UTC conversion — independently in both files (matching the existing pattern of each file having its own local date-range helper rather than a shared cross-file import, since `ai-query.service.ts` already imports *from* `ai-vertical-templates.service.ts` and a reverse import would be circular). Replaced all ~36 call sites across both files. This fixes every pre-existing template's date defaulting too, not just the new ones — a genuine correctness improvement to already-shipped functionality, found as a side effect of testing new code, not sought out separately.

Two of the new regression tests were themselves initially "tautological" — they computed their expected value with the identical buggy formula the implementation used, so they would have passed even with the bug present. Caught during the same pass and fixed by re-deriving expected values with the corrected local-getter formatting.

## Addendum 8 (2026-07-15) — full live UAT of all 70 new templates: 3 real bugs found and fixed

Founder asked for UAT of all 110 templates, explicitly scoped down (with founder agreement) to the 70 templates added in Addendum 7 — the original 40 already have live-verification history across Addendums 2-6. Built a reusable live UAT harness (`tests/e2e/uat-70-templates.js`) that seeds real fixtures via direct SQL for every business type under test, then calls the real `window.api.ai.query` IPC handler for all 70 questions — the same production code path a real user's click triggers, real classification, real DB, real answer formatting. Not a mock.

**Bug 1 (severe, pre-existing, affects the ORIGINAL templates too) — model chat session never reset, causing escalating multi-minute latency.** `NodeLlamaProvider`'s `LlamaChatSession` is created once and reused for the app's entire lifetime; by default it appends every prompt+response to its own history and re-evaluates the full accumulated context on every call. Every question this assistant answers is independent by design (Section 5.6: "no multi-turn chat memory... every question is independent"), but the session object was never told that. Measured live: the first ~8 questions in a session took the expected ~65-85s each; the 9th-10th (once accumulated history approached the 4096-token context cap) took 452s and 463s — a 6x+ spike from context-shift/recompute cost that would only keep recurring for any real user asking more than ~8 questions without restarting the app. Fixed with `session.resetChatHistory()` after every exchange in both `classifyIntent()` and `generateResponse()`. Confirmed via direct before/after measurement: every question after the first now takes 6-9 seconds, not 65-450+.

**Bug 2 (severe) — classification accuracy had silently degraded to a 50% miss rate.** The full 70-question live battery initially found 35 of 70 misrouted — the classification grammar now has to choose among ~110-130 template names (static + active vertical) instead of the original ~20-40, and the 1.5B model's discrimination ability doesn't scale with catalog size. Fixed with 40 new deterministic `FAST_PATH_PATTERNS` entries (same proven technique already used for the original catalog — see the file's own header comment), including two placement-order fixes (`inventory.topSellingByQuantity` had to be checked before the broader pre-existing `inventory.topRevenueProducts` pattern; same for `sales.walkInVsRegistered` vs `sales.totalThisMonth`, and `finance.profitTrend` vs `finance.profitAndLoss`'s bare `/\bprofit\b/i`) and three phrasing-adjacency fixes (`\s+` between a trigger word and its qualifier assumed no words in between — real phrasing like "vaccinations **are** due" or "shoots **do I have** coming up" broke that assumption). Re-verified live after each round: 35 wrong → 5 wrong → 0 wrong across three full 70-question passes. Five of the original 35 were being refused outright (classified as `out_of_scope`) rather than just misrouted — the worst failure mode, since the question was genuinely answerable.

**Bug 3 (in the UAT harness itself, not the app) — seed data silently invisible to date-range queries.** The harness's own SQL seed data stored dates as ISO text strings (`.toISOString()`, or the SQL `CURRENT_TIMESTAMP` literal); a fresh Prisma-written row was confirmed to store the same columns as INTEGER (Unix epoch milliseconds) instead. A text-stored date compared against an integer-bound date-range filter — exactly what every template here does with a real JS `Date` object — silently fails to match, which is why `vet.vaccinationsDue` returned the empty-fallback message despite a correctly-dated `VaccinationRecord` existing. Not a product bug, but it meant an earlier "70/70 passing" result wasn't fully trustworthy — some date-filtered vertical templates could have been silently untested rather than confirmed working. Fixed with a `fixDateColumnTypes()` post-processing pass (`CAST(strftime('%s', col) AS INTEGER) * 1000`, guarded by `typeof(col) = 'text'` so it only ever touches the harness's own malformed rows, never genuine app data) covering every date column the harness sets across all 70 templates' fixtures. Re-ran the full battery after the fix and manually verified the answer text for every date-filtered vertical template (vet, dental, site visits, shoots, events, tailoring, pest control, driving tests, legal hearings, ROC filings, supplier lead time, staff attendance, etc.) — every one now correctly finds and reports its seeded record with the right name, date, and count.
