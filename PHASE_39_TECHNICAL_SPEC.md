**App icon** (`resources/icon.ico`/`icon.png`) is generated procedurally by `scripts/generate-icons.js` — pure-JS pixel manipulation drawing a gradient square + hand-drawn "S" glyph bitmap, explicitly documented in its own header comment as "a good-faith interim brand asset... replace before a customer-facing release." This is exactly what Phase 39 does.

**Print template footers** (`print.service.ts`) — plain text "Generated by Sarang Business OS Lite | Aszurex | www.aszurex.com" on invoice, receipt (thermal, with QR), and quotation templates (at least 3 confirmed instances; audit found the pattern via grep, not yet an exhaustive per-template list — confirm full list during implementation). **Two separate, drifted About experiences found** — this needs a decision at this spec's sign-off, not a silent pick:
- `AboutScreen.tsx`, reachable at route `/about` (also in `Sidebar.tsx`'s `NAV_ITEMS`, label "About"): richer — business info card, privacy/data facts, data storage paths, legal disclaimer, support links, footer copyright.
- `AboutSection()` inside `SettingsScreen.tsx`, reachable via Settings → About tab: simpler — identity block, five trust bullet points, a "Check for Updates" button, and a generic Aszurex sales-pitch box.
- These currently show **different content** for the same conceptual "About Sarang" destination depending on how a user gets there. Recommend consolidating into one real screen (`AboutScreen.tsx`'s richer shape is the better base — it already has the update-checker missing from it today, which should be merged in) and making Settings' "About" tab either link out to `/about` or be removed as a separate destination. This is where the founder's requested authentic-voice redesign (Section 0 backstory) lands.

**New brand assets, staged at `resources/branding-v2/`, all confirmed genuine transparent PNGs** (80-93% transparent pixels, real alpha channel, not opaque images with a baked-in background — verified by direct pixel inspection, not assumed from file properties):

| File | Canvas | Content bounding box | Content aspect ratio |
|---|---|---|---|
| `sarang-wordmark-lockup.png` | 1536×1024 | x:322-1232, y:342-621 → 911×280 | 3.25:1 (very wide) |
| `sarang-icon-mark.png` | 1536×1024 | x:421-1116, y:215-794 → 696×580 | 1.20:1 (near-square) |
| `aszurex-partnership-mark.png` | 1536×1024 | x:415-1126, y:339-653 → 712×315 | 2.26:1 |

All three need cropping to their content bounding box before use (the surrounding canvas is mostly empty transparent padding — using the full 1536×1024 canvas as-is would waste enormous space in any UI slot and force awkward scaling).

**Color tokens** already exist in `tailwind.config.ts`: `brand` (DEFAULT `#00AEEF`, full 50-900 ramp), `dark #0F172A`, `success #22C55E`, `warning #F59E0B`, `danger #EF4444`, `surface #F8FAFC`, plus shadcn-style semantic HSL tokens (`background`, `card`, `foreground`, `primary`, `muted`, `border`, `info`) driven by CSS variables in `globals.css` for light/dark mode. This is the single source of truth to extend, not replace — Phase 39 adds/adjusts values here, it doesn't introduce a second palette system.

**CSP consideration**: `splash.html` has `<meta http-equiv="Content-Security-Policy" content="default-src 'none'; style-src 'unsafe-inline';">` — embedding the new logo as an image requires either an inline SVG (no CSP change needed) or a base64 `data:` URI with `img-src data:` added to the CSP (a scoped, reviewable addition — still blocks all network image loads, just permits a locally-embedded asset). Recommend checking whether converting the PNGs to SVG is feasible first (crisper at all sizes, no CSP change); if not, the `data:` URI + CSP addition is the fallback.

## 2 — DECISIONS NEEDING SIGN-OFF BEFORE IMPLEMENTATION

**(a) Sidebar top-left branding — literal wordmark lockup vs. compact icon + text.** The founder's instruction was explicit: the wordmark lockup (`sarang-wordmark-lockup.png`) is "top left corner branding across Sarang." But its content is 3.25:1 and 911px wide at source resolution — the Sidebar's current logo slot is a 36×36px icon next to a text block, in a sidebar that's 280px wide expanded / 72px collapsed. Fitting the full lockup (with its small "Your Business. Your Way." tagline sub-text) there either means: **(i)** restructure the Sidebar header to be taller/wider to fit a legible cropped wordmark (dropping the separate dynamic business-name line, or stacking it below instead of beside), or **(ii)** use the wordmark lockup at genuine hero moments only (splash, login, setup wizard, About screen) where there's room, and use the compact `sarang-icon-mark.png` + the existing dynamic business-name text for the Sidebar's persistent 36px slot specifically, since it's a functionally different (small, always-visible, business-identifying) context than a hero placement. **Recommend (ii)** — it keeps the business-name identification that's functionally necessary in a multi-business/multi-profile future and avoids shrinking the wordmark to illegibility, while still honoring "wordmark lockup = the app's primary top-left brand identity" everywhere it actually fits well. Confirm before implementation.

**(b) About page consolidation.** Recommend merging into one screen at `/about` (Section 1) rather than maintaining two. Confirm before implementation — this changes navigation (Settings' About tab either redirects or is removed).

**(c) SVG vs. PNG for the new assets.** If the founder has (or can generate) SVG versions of the three marks, that avoids the CSP change in `splash.html` and gives crisp rendering at every size (icon mark scaled to 16px for a favicon looks very different in quality from an SVG vs. a downscaled 1536px PNG). If only PNGs exist, Phase 39 proceeds with cropped PNGs + the scoped CSP addition. Confirm which is available before implementation starts.

**(d) `sharp` as a new devDependency for icon generation.** `scripts/generate-icons.js` currently draws icons with zero dependencies (raw pixel manipulation) specifically to avoid adding one. Producing a proper multi-size ICO from a real source PNG (resizing `sarang-icon-mark.png`'s cropped content into 16/32/48/64/128/256px squares with correct anti-aliasing) is impractical to hand-roll to production quality. Recommend adding `sharp` (MIT license, free, build-time-only devDependency — never ships inside the packaged app, so it doesn't affect installer size or the zero-cost/offline runtime guarantee) to replace the procedural glyph-drawing in `generate-icons.js` with real image resizing from the source asset. Confirm this dependency addition is acceptable (it's free and dev-only, but is a real new tool in the build chain).
