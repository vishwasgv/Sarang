# Phase 54F.17 — GST Filing Support: HSN Summary + GSTR-3B Reconciliation Preview — Technical Spec

## 1. Audit findings (confirmed against current code, and against the current GST portal, 2026-07-09)

**Current code.** `generateGSTR1()` (`report.service.ts:750-811`) builds B2B (customer has a GSTIN) and B2CS (retail, grouped by place-of-supply+rate) sections from real invoice-item data, correctly split IGST vs. CGST/SGST via `Invoice.gstType`. It has no HSN-wise aggregation and no GSTR-3B builder at all — confirmed by grepping `report.service.ts` for `hsn`/`HSN`/`3B`/`3b`, no matches outside `Product.hsnCode`/`InvoiceItem.hsnCode` themselves. `InvoiceItem.hsnCode` (`schema.prisma:468`) **is already captured on every invoice line** — confirmed populated at creation time in `billing.service.ts:192,294` (snapshotted from `Product.hsnCode` at the moment of sale, not looked up live later) — so an HSN summary needs zero new schema, purely aggregation over data that already exists. `ReportsScreen.tsx`'s existing `category: 'gst'` reports are hidden entirely unless `taxModel === 'GST'` (`ReportsScreen.tsx:993`) — the exact gate this phase's 2 new reports need, already proven by the existing GSTR-1 tile. Existing GSTR-1 report already carries an honest disclaimer ("This report is for reference only. Verify with a CA before filing on the GST portal.") — the precedent this phase's tone should match exactly, not invent a new one.

**Current GST portal mechanics, verified via web research before writing this spec** (per the master plan's own instruction — "GST return schemas are precise and change with notifications; get the exact current schema confirmed before building, don't guess"):
- **Table 12 (HSN-wise summary) is now dropdown-only entry on the government portal itself.** Per GSTN's own Phase-3 rollout (effective from the May 2025 tax period), HSN codes in Table 12 must be selected from a portal-provided master list, not typed or bulk-uploaded as free text — split into 12A (B2B, mandatory) and 12B (B2C, optional under ₹5cr turnover). A literal "generate a JSON matching the old free-text HSN summary schema" feature would be promising something the portal may not actually ingest the way it used to.
- **Table 13 (document-wise summary — invoice/CN/DN series, counts, cancelled counts) is now also mandatory**, filed alongside Table 12 under the same Phase-3 rollout.
- **HSN digit requirement**: 4-digit HSN mandatory on B2B invoices (optional on B2C) for turnover up to ₹5cr in the prior year; 6-digit mandatory on all invoices above ₹5cr. This app has no turnover-tier awareness today — a report should surface whatever digit-length the owner's own `Product.hsnCode` values actually contain, not assume or enforce a tier.
- **GSTR-3B Tables 3.1 and 3.2 (outward supplies) are now non-editable, auto-populated from an already-filed GSTR-1/1A/IFF**, as of the November 2025 tax period onward. A taxpayer can no longer manually key outward-supply figures into GSTR-3B — the portal fills them in from GSTR-1 and locks them. This means a "GSTR-3B summary the owner fills into the return" framing is now inaccurate; the honest, useful shape is a **reconciliation preview** — "here is what should appear in your auto-populated 3B, based on what Sarang has recorded, so you can catch a mismatch before or after filing GSTR-1" — not a form to copy numbers from.
- **Table 4 (Input Tax Credit) remains genuinely out of reach**: it requires purchase-side GST tracking with supplier-GSTIN matching, credit-eligibility rules, and reversal tracking that this app does not model at that granularity today (`purchaseOrder.service.ts`/`Expense` track cost and vendor, not ITC-specific ledger fields). Building it is a materially larger, separate feature, and edges toward the "accounting software" positioning this project has deliberately stayed out of.
- **Founder decision recorded** (see the F.17 AskUserQuestion, 2026-07-09): given the above, this phase builds accurate **reference reports** (HSN summary, document-wise summary, GSTR-3B outward-supply reconciliation preview) that the owner or their CA uses alongside the government portal — not a claimed portal-upload JSON file, since the portal's actual current ingestion mechanism for these specific tables (dropdown-only HSN selection; auto-populated, locked 3B outward-supply tables) may no longer accept a bulk file the way the older offline utility did, and any JSON this app produced would need re-verification against the live portal before real use regardless — a report the owner reads and manually cross-checks is the honest, durably-useful deliverable.

## 2. Scope to deliver

**2.1 — HSN-Wise Summary Report.**
New `generateHSNSummaryReport({ dateFrom, dateTo })` in `report.service.ts`, reusing the exact same invoice-item query shape `generateGSTR1` already uses (same date-range filter, same `status: { not: 'CANCELLED' }` exclusion — cancelled invoices never belonged in a filing in the first place). Groups by `(hsnCode, taxRate)`, matching the real Table 12 shape: HSN code, description (from `Product.productName`/category if the HSN itself has no separate description field — confirmed `Product` has no dedicated HSN-description field, so the aggregation uses the product name as the closest available description, flagged in the report's own copy as "description sourced from product name, not an official HSN description lookup"), UQC (unit — reused from `InvoiceItem`'s existing `weightUnit`/product `unit` field), total quantity, total value, taxable value, IGST/CGST/SGST amounts. **Split into B2B and B2C sub-tables** (mirroring the real Table 12A/12B split) using the same `customer.taxNumber` presence check `generateGSTR1` already uses to separate B2B from B2CS. Rows with a missing/blank `hsnCode` are grouped under an explicit "No HSN Code" bucket rather than silently dropped or crashing — surfaces a real data-quality gap (a product missing its HSN code) instead of hiding it.

**2.2 — Document-Wise Summary Report (Table 13 shape).**
New `generateDocumentSummaryReport({ dateFrom, dateTo })`: for each document series already tracked by this app's atomic-numbering system — Invoice, CreditNote, DebitNote — reports the number series prefix, the from/to document numbers issued in the period, total count issued, and cancelled count (`Invoice.status === 'CANCELLED'`; `CreditNote`/`DebitNote` don't currently have a cancellation concept, confirmed by checking their schemas — reported as 0 cancelled for those two, not fabricated). This is a small, cheap addition (reuses number-sequence data that already exists per F.1's atomic-numbering fix) that closes the other half of what GSTN's current Phase-3 rollout actually requires alongside the HSN summary.

**2.3 — GSTR-3B Outward-Supply Reconciliation Preview.**
New `generateGSTR3BPreview({ dateFrom, dateTo })`: computes what Table 3.1 (taxable/zero-rated/nil-rated/exempt outward supplies, reverse-charge inward — this app has no reverse-charge purchase tracking, so that line reports as "Not tracked by Sarang — enter manually if applicable," an honest gap disclosure rather than a fabricated zero) and Table 3.2 (inter-state supplies to unregistered persons/composition dealers, broken down by destination state — reuses `Invoice.buyerState`/`gstType === 'IGST'` the same way `generateGSTR1`'s B2CS section already groups by `placeOfSupply`) *should* show, computed from the exact same underlying invoice data GSTR-1 itself is built from — so if this preview's numbers don't match what the portal auto-populates from the owner's actually-filed GSTR-1, that itself is the useful signal (either GSTR-1 wasn't filed for this period, or something in Sarang's records diverges from what got filed). Table 5 (exempt/nil-rated/non-GST outward supplies) is covered by the same taxable-value aggregation at `taxRate === 0`. **Table 4 (ITC) is explicitly out of scope** — the report includes a clear "Input Tax Credit (Table 4) is not covered — this app does not track purchase-side GST input credit" note rather than a blank or misleading zero.

**2.4 — Reports UI.**
Two new `ReportType` entries (`hsnSummary`, `gstr3bPreview`) plus the existing `documentSummary` naming convention, all registered in `REPORT_DEF_META` with `category: 'gst'` (inherits the existing `taxModel === 'GST'` visibility gate automatically, no new gating code needed) and `permission: 'reports.tax'` (same permission the GSTR-1 tile already uses). Each gets: a fetch case, a CSV export case, a summary-cards case, a render-view case — following the exact 5-switch-statement extension pattern this bundle's Section 7A already established for Test Score/Compliance Task reports. Each carries the same disclaimer style as the existing GSTR-1 tile, adapted per report (e.g. HSN Summary: *"This report is for reference only — HSN codes must be selected from the government portal's own list when filing. Verify with a CA before filing."*). Full 12-language translation for the new strings, since GST-model businesses exist in every locale this app supports (not an English-locked vertical) — same standard as F.15's dashboard/audit strings, not the placeholder-copy shortcut used for the English-locked coaching/CA-firm reports in Section 7A.

**2.5 — PDF/print + CSV export.**
Both new reports get the same PDF-via-Chromium-print pipeline and CSV export every other report already has — no new export mechanism.

## 3. Explicitly out of scope

- **No JSON file claiming portal-upload compatibility** — see Section 1's founder decision. If GSTN's offline-utility JSON format for these tables becomes clearly re-verifiable and stable in the future, that's a founder-scoped follow-up, not assumed here.
- **No Input Tax Credit (Table 4) computation** — needs purchase-side GST tracking this app doesn't have; explicitly disclosed as not covered in the report output itself (2.3), not silently blank.
- **No reverse-charge inward-supply tracking** — same reasoning, explicitly disclosed rather than fabricated.
- **No auto-submission to any government API** — this app makes zero outbound network calls by design (`[[project_audit_findings_2026_07]]`'s Phase-53 finding); these reports are read/print/export only, exactly like every other report in this module.
- **No turnover-tier-aware HSN digit enforcement** (4-digit vs 6-digit mandate based on the business's own turnover) — the report reflects whatever `hsnCode` values are actually on file; validating them against the business's turnover tier is a separate, not-yet-requested feature.
- **No changes to the existing `generateGSTR1`** — B2B/B2CS logic is already correct and unaffected; this phase adds two new report types alongside it, not a rewrite.

## 4. Testing plan

- Live: launch the real dev app under a `taxModel === 'GST'` business (e.g. switch to a fresh business type if the dev DB isn't already GST), confirm all 3 new report tiles appear only when `taxModel === 'GST'` (and disappear when switched to a non-GST business, confirming the existing gate correctly covers the new tiles with zero new gating code), generate each report against real seeded invoice data, confirm the HSN summary total taxable value reconciles against the existing GSTR-1 report's own totals for the same date range (a genuine cross-report correctness check, not just "does it render").
