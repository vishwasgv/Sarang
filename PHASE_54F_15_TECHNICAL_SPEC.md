# Phase 54F.15 — Security Baseline: Password Policy, DB Encryption-at-Rest, Tamper-Evident Audit Log — Technical Spec

## 1. Audit findings (confirmed against current code, 2026-07-09)

**Password policy — two independent, drifted enforcement points, not one.**
- `setup.service.ts:158` seeds `password_min_length` = `'6'`. `auth.service.ts:212` (`changePassword`) and `users.handler.ts:71` (`adminResetPassword`) both read this Setting at call time and enforce it.
- `users.validation.ts:6` (`CreateUserSchema`, used by `users:create`) hardcodes `z.string().min(6, ...)` — it does **not** read `password_min_length` at all. Raising the Setting today would tighten password-change/reset but silently leave new-user creation stuck at 6. This is a real, confirmed drift bug, not a hypothetical.
- No complexity requirement exists anywhere (length only — no uppercase/digit/symbol check), and the master plan doesn't ask for one; out of scope here, noted for completeness.
- **No Settings UI exposes `password_min_length` at all** (grepped the renderer — zero matches). It's a backend-only value nobody can currently change without direct DB access. Same class of gap already flagged (and left as a follow-up, not fixed) for `allow_negative_inventory`/`allow_expired_batch_sale` in F.2's completion note.

**Audit log — confirmed exactly as the master plan describes.**
- `AuditLog` (`schema.prisma:577-594`) has no hash/chain column of any kind — `id`, `userId`, `action`, `entityType`, `entityId`, `oldValue`, `newValue`, `createdAt`. Anyone with file access to `sarang.db` can edit or delete rows with a plain SQLite client and nothing would ever notice.
- `audit.service.ts:33-47`'s `logAction` wraps its insert in a bare `try/catch` that only does `console.error(...)` on failure — callers never check the return value (it's fire-and-forget everywhere, e.g. `await auditService.logAction(...)` immediately after `db.user.create(...)` in `users.handler.ts:29`, not inside the same transaction). A failed audit write is currently invisible to everyone except someone tailing the Electron main-process console.
- The existing "surface a silently-missed thing on the Dashboard" pattern is already established and reusable: `analytics.service.ts`'s `DashboardAlert` union (`NO_BACKUP`, `PENDING_REMINDERS`, `LOW_STOCK`, `LARGE_OUTSTANDING`) is exactly this shape — F.5 and F.8 both extended it rather than inventing a new mechanism.
- `pruneOldAuditLogs()` already exists (730-day default retention) — any hash-chain design has to stay correct under pruning (chain-from-genesis breaks the moment old rows are deleted), so the chain cannot assume it always reaches row 1.

**DB encryption at rest — genuinely the largest open question, confirmed against the actual dependency tree.**
- `prisma/schema.prisma:5-8`: `datasource db { provider = "sqlite" }` using the default `prisma-client-js` generator — Prisma's bundled Rust query engine talks to plain SQLite directly. It has no cipher support; encrypting the file would require moving off the default engine entirely.
- Confirmed via `package.json`: **this project currently has zero native (compiled) dependencies.** `bcryptjs` (pure JS) was deliberately chosen over `bcrypt` (native) for exactly this reason. There is no `electron-rebuild`/`@electron/rebuild` step anywhere in the build pipeline, and `electron-builder`'s config has no `asarUnpack`/native-module handling set up.
- Every realistic SQLCipher-compatible option for Node/Electron (`better-sqlite3-multiple-ciphers`, `@journeyapps/sqlcipher`, `node-sqlcipher`) is a **native addon** — encrypting the DB file would be this project's first native dependency ever, which means: (a) a rebuild step keyed to Electron's exact ABI/Node version must be added to the build pipeline (currently absent), (b) `electron-builder`'s `asarUnpack` must be configured so the compiled `.node` file isn't sealed inside the (unreadable-to-native-code) asar archive, (c) anyone building this project from source now needs a working native toolchain (Python + MSVC Build Tools on Windows, Xcode CLT on Mac) instead of just `npm install`, and (d) Prisma would need its `driverAdapters` preview feature turned on and every `getPrisma()` call site's client construction changed to route through `@prisma/adapter-better-sqlite3` (or equivalent) instead of the default engine — a real, if mechanical, migration, not a config flag flip.
- This is a materially bigger, riskier change than F.1-F.14's scope: it's the first crack in a build-simplicity invariant this project has held since its start (zero-cost, zero-native-deps, `npm install && npm run dev` always just works), for a single founder maintaining this alone. That tradeoff is exactly what Section 0 of the master plan says needs "build/packaging compatibility with electron-builder... confirmed before committing" — i.e. a real go/no-go call, not something to default into silently.
- **The practical alternative already available today, zero-cost, zero-native-deps**: OS-level full-disk encryption (Windows BitLocker, macOS FileVault, Linux LUKS) already protects `sarang.db` at rest the moment the machine is powered off or the disk is removed — the exact threat model "DB encryption at rest" is normally defending against for a single-machine offline desktop app. It doesn't defend against a live, logged-in, unlocked machine (neither would DB-level encryption alone, in practice — a running app must hold the decryption key in memory to serve any query, so a compromised-while-unlocked machine is already lost either way). This is genuinely a scope decision with a real cost/benefit tradeoff, not a "just do the more thorough thing" call — see Section 3 for how it's being resolved.

## 2. Scope to deliver

**2.1 — Password policy: raise the default, and close the drift (S, no open decision).**
- Raise `setup.service.ts`'s seeded `password_min_length` default from `'6'` to `'10'`.
- Fix `CreateUserSchema` (`users.validation.ts`) to stop hardcoding `.min(6, ...)`. Since Zod schema construction happens at module load (before any DB call), move the length check out of the static schema and into the `users:create` handler itself — read `password_min_length` the same way `changePassword`/`adminResetPassword` already do, immediately after the (now length-free) Zod parse succeeds, so all three password-setting paths enforce the *same*, live-configurable minimum instead of two independent hardcoded/Setting-driven values that can drift again.
- Add a **Settings UI control** for `password_min_length` (Settings → Security, a new or existing section — check `SettingsScreen.tsx`'s `SECTIONS` array for the right home) so an owner can actually change it, closing the "backend-only Setting nobody can reach" gap noted in Section 1 — this was flagged as a real, user-facing gap by the audit that reads this Setting, not a scope-creep add.
- Existing users are **not** forced to change their password retroactively when the minimum rises (no forced-reset flow) — same "additive, non-disruptive" convention this project uses for every other default-value change (e.g. F.5's `auto_backup_enabled` flip). A stricter minimum only applies going forward, on the next password change/creation.

**2.2 — Tamper-evident audit log via hash chaining (M, no open decision).**
- Add two nullable columns to `AuditLog`: `hash` (`String?`, the SHA-256 hex digest of this row's own content plus the previous row's hash) and `prevHash` (`String?`, copy of the immediately-prior row's `hash`, or `null` for a genesis row). Nullable so existing pre-migration rows don't need a backfill guess at a hash they were never designed to have — the chain simply starts fresh from the first row written after this migration ships; the old, unchained rows stay exactly as they are (a boundary, not a false claim about their integrity).
- `logAction` is rewritten to: (1) build the exact `createdAt` timestamp itself (`new Date()`) rather than relying on Prisma's `@default(now())`, so the same value is both stored and hashed; (2) inside a single `db.$transaction`, read the most recent row's `hash` (ordered by `createdAt desc`, `id desc` as a tiebreaker), compute `hash = sha256(prevHash + action + entityType + entityId + oldValue + newValue + createdAt + userId)` (Node's built-in `crypto`, no new dependency), and insert the new row with both `hash` and `prevHash` set. This reuses the exact same "claim the latest state inside one transaction, rely on SQLite's write-transaction serialization" pattern `sequence.service.ts` already established for numbering — the same guarantee, applied to a hash chain instead of a counter.
- New `verifyAuditLogChain()` in `audit.service.ts`: walks the table in insertion order, recomputes each row's expected hash from its own content plus the previous row's *stored* hash, and returns the first row where the recomputed hash doesn't match (tampered/edited row) or where `prevHash` doesn't match the prior row's actual `hash` (deleted/reordered row) — or a clean "N rows verified, chain intact" result. Deliberately **on-demand, not automatic on every launch** — a "Verify Integrity" button on the existing Audit Log screen, not a background job, because walking a potentially large table is real work and this project's own convention (F.8, F.10) is to prefer a manual click over unnecessary background cost for anything that isn't itself time-sensitive.
- `pruneOldAuditLogs()` needs one adjustment: after deleting old rows, the chain's first surviving row now has a `prevHash` pointing at a row that no longer exists. `verifyAuditLogChain()` handles this by treating the oldest surviving row as a new valid "genesis" for verification purposes (its own `prevHash` is trusted as a boundary, not re-derived) — otherwise every routine prune would immediately look like tampering, which would make the feature cry wolf and get ignored. This is a real design point, not an afterthought: flagged explicitly so a reviewer doesn't mistake it for a loophole (an attacker who deletes recent rows AND the audit_log_retention Setting's cutoff is a threat model this chain cannot and does not claim to catch — it defends against silent row edits/deletes, not "delete the whole tail and claim nothing happened before the new genesis").
- `logAction`'s failure path changes from silent `console.error` to also writing a Setting flag (`audit_log_last_failure_at`) on catch, and `getDashboardAlerts()` gains a new `AUDIT_LOG_FAILURE` alert type (same file, same pattern F.5/F.8/F.9 already extended) that fires if that flag is set and clears once a subsequent `logAction` call succeeds — reusing the exact mechanism already on the Dashboard rather than a new notification channel.

**2.3 — DB encryption at rest: recommended default + explicitly optional stretch (see Section 3 for the founder decision this section depends on).**
- **Default delivered in this pass regardless of the encryption-path decision**: a one-time, dismissible onboarding/Settings notice recommending the owner enable their OS's built-in disk encryption (BitLocker/FileVault/LUKS, whichever `process.platform` indicates), with a short explanation of why (protects the data if the laptop is lost or stolen while powered off) — zero-cost, zero-native-deps, ships immediately, and is the actual correct mitigation for this app's real threat model (single-machine offline desktop app, not a multi-tenant server).
- **The native-driver DB-level encryption path is not started in this pass** — it needs the founder's explicit go/no-go first, per Section 3, given it's the first native dependency this project would ever take on and a real build-pipeline change. Flagged the same way F.18 (multi-location) was flagged in the master plan: a real, scoped-out decision point, not a silently dropped feature.

## 3. Open decision requiring founder sign-off before 2.3's native-driver path can start

Two honest options for "real" (not just OS-level) database encryption at rest:

- **Option A — OS-level disk encryption only** (recommended default, already scoped into 2.3 above regardless of this decision): rely on BitLocker/FileVault/LUKS, ship the onboarding nudge, stop there. Zero build risk, zero native dependency, ships today. Leaves the DB file readable in plaintext to anything running on an *unlocked* machine — the same exposure DB-level encryption alone would also have, since a live app needs the key in memory to serve queries either way.
- **Option B — Add a native SQLCipher-compatible driver** (`better-sqlite3-multiple-ciphers` + Prisma's `driverAdapters` preview feature): genuinely encrypts the file on disk independent of OS-level encryption, but is a real, ongoing maintenance cost — first native dependency ever, new rebuild step, `asarUnpack` config, a working native toolchain now required to build from source, and Prisma driver-adapter migration touching how `getPrisma()` constructs its client.

This is asked, not decided here, because it's a genuine tradeoff between two real things the founder has said matter — the zero-cost/offline/simple-to-build identity of this project vs. defense-in-depth on data at rest — and picking wrong either over-builds (native-module maintenance burden neither the founder nor a future contributor signed up for) or under-builds (leaving a real audit finding unaddressed). Recorded as its own explicit decision point rather than defaulting to either extreme, per this project's own established rule (`[[feedback_scope_and_testing]]`): don't assume the most expensive interpretation, but don't silently drop a real finding either — ask.

**Decision (founder, 2026-07-09): Option A — OS-level disk encryption only.** Native SQLCipher-compatible DB-level encryption (Option B) is explicitly not being built in this pass, for the reasons above — it stays an option to revisit later if the product's threat model changes (e.g. a future multi-machine sync feature would change this calculus), not a silently dropped item.

## 3A. Scope discovery during implementation: most AuditLog writes bypass `logAction` entirely

Confirmed while building 2.2: `logAction` is not the only write path to `AuditLog`. ~150 call sites across 54 service files write audit rows directly (`db.auditLog.create({ data: {...} }).catch(() => {})`, e.g. `driving.service.ts` alone has 8), never touching `logAction` at all — these rows would stay completely outside any hash chain built inside `audit.service.ts`, no matter how that function is implemented. 3 further call sites (`appointment.service.ts` ×2, `setup.service.ts` ×1) write via `tx.auditLog.create` *inside an already-open business transaction* — these specifically cannot be safely converted to call `logAction` (which opens its own `$transaction`) without risking a nested-transaction deadlock on SQLite, a separate, harder problem than the other 54 files.

**Decision (founder, 2026-07-09, "do whatever is best"): ship the hash chain scoped to `logAction`, document the rest as a follow-up, do not attempt the 54-file migration in this pass.** Reasoning: `logAction` already covers this app's actual security-sensitive event stream — login, password changes, user creation, role/permission changes (`users.handler.ts`, `auth.service.ts`) — which is what an auditor or the founder would actually reach for if investigating "did someone tamper with who has access." The ~150 bypassing call sites are almost entirely routine entity CRUD logging ("DrivingVehicle created", "StudentTestScore updated") — real, but a different and lower-stakes category. Migrating all 54 files mechanically in one pass (plus updating the ~5 test files whose mocks assert on the old direct call) is meaningfully larger than F.15's original "S/M" sizing for this item, and doing it as a single sweeping find-and-replace without the kind of per-file review this codebase's other phases have consistently gotten (see e.g. F.4/F.7's deliberate scope narrowing to avoid an under-reviewed sweeping change) is a real regression risk, not just extra effort. This is being recorded as an explicit, named boundary — the same way F.18 was flagged rather than silently started or silently dropped — not hidden inside the completion note as if the chain covers "every mutation" the way `AuditLog`'s own pre-existing schema comment claims. **Follow-up candidate**: a dedicated future pass to migrate the 54 direct-write files to `logAction`, with real per-file review given the volume, plus a design decision on how (or whether) to safely chain the 3 in-transaction call sites too.

## 4. Explicitly out of scope

- Password complexity rules (uppercase/digit/symbol) — the master plan and this audit only flagged length; adding complexity rules is a separate, undiscussed UX tradeoff (forced-complexity policies are known to sometimes push users toward *weaker* patterns like `Passw0rd!`) and isn't being bundled in here.
- Forced password rotation / expiry — not requested, not evaluated here.
- Any change to `bcryptjs`'s hashing algorithm or `SALT_ROUNDS` — orthogonal to this spec's three items and already adequate (12 rounds, confirmed in `auth.service.ts`).
- Retroactively hash-chaining pre-migration `AuditLog` rows — see 2.2, the chain starts clean from the migration point forward, not backfilled with a fabricated chain over historical data that never had one.
- Encrypting anything other than the SQLite file itself (e.g. backup `.zip` files, exported CSV/PDF reports) — a separate surface, not in this audit finding, not addressed here.

## 5. Testing plan

- Unit: `users.validation.ts`/`users.handler.ts` (new-user creation respects the live `password_min_length` Setting, not a hardcoded 6; rejects below-minimum on all three paths — create, change, admin-reset — with the same effective minimum); `audit.service.ts` (hash chain: a fresh genesis row, a multi-row chain verifies clean, a tampered row's edited field is detected, a deleted-row gap is detected, pruning doesn't false-positive against the new oldest-surviving-row boundary, `logAction` failure sets and a subsequent success clears the dashboard-alert flag); `analytics.service.ts` (`AUDIT_LOG_FAILURE` alert appears/clears correctly).
- Live: launch the real dev app, confirm the new password-minimum Settings control actually changes enforcement on a real create/change/reset attempt through the UI (not just the API); confirm the Audit Log screen's new "Verify Integrity" button reports a clean chain against real seeded data, then confirms it detects a hand-tampered row (direct SQL edit via `node:sqlite` against the dev DB, same tooling already used for this session's migrations); confirm the OS-encryption onboarding notice renders and is dismissible.
