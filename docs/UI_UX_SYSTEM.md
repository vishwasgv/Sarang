## COMPONENT INTERACTION STATE GUIDE

Every interactive component must implement all applicable states:

| State | When | Visual Signal |
|-------|------|--------------|
| Default | At rest | Base colors, normal cursor |
| Hover | Mouse over | Subtle background lighten/darken |
| Focus | Keyboard Tab / click focus | Blue ring `ring-2 ring-[#00AEEF]` |
| Active | Mouse held down | Pressed appearance (slightly darker) |
| Disabled | `disabled` prop / no permission | `opacity-50`, `cursor-not-allowed`, no hover effect |
| Error | Validation failure | Red border `border-red-400`, red helper text |
| Loading | Async operation | Spinner or skeleton, pointer events none |
| Success | Completed action | Green confirmation state (transient, 2s) |

## EMPTY STATE DESIGN RULES

Every list, table, or data view must define its empty state.

Empty states must contain:
1. An icon (Lucide, relevant to the content type)
2. A title: "No [items] yet"
3. A description: Brief explanation why and what to do
4. An action button: "Add your first [item]"

Never show:
- A blank white space
- Raw "No data" text without context
- A loading spinner that never resolves

## UX DECISION RULES

**When to use Modal vs Page navigation:**
- Use Modal: Quick actions, confirmations, simple forms (≤ 5 fields)
- Use Page: Complex forms, multi-section data entry, full record detail

**When to use Toast vs Modal:**
- Use Toast: Success confirmations, non-critical warnings, info messages
- Use Modal: Actions that require explicit confirmation before proceeding (delete, restore)

**When to show a spinner vs skeleton:**
- Use Skeleton: Page/section loads (initial data fetch)
- Use Button spinner: Form submission, single action in progress
