# Phase 48 — Clothing & Tailoring Vertical Depth: Technical Spec

## 1. Audit findings (confirmed against current code)

- **`Product`/`ProductVariant`**: confirmed zero gender concept anywhere. `ProductVariant.size`/`.color` are free-text (`schema.prisma:713-714`). `variant_tracking` module (`industry-template.service.ts:123-124`) is assigned by default to exactly `CLOTHING`/`FOOTWEAR` and no other business type — the correct, already-existing config flag to gate a new gender selector on, matching the project's "config flags only" rule (no new flag needed).
- **`TailoringOrder`** (`schema.prisma:2649-2679`): `garmentType` already includes `SHIRT|PANT|SUIT|KURTA|SALWAR_KAMEEZ|BLOUSE|LEHENGA|SAREE_BLOUSE|JACKET|OTHER` — no gender or Indian/Western style field exists.
- **`MeasurementRecord`** (`schema.prisma:2622-2647`): 10 unisex fields (chest/waist/hips/shoulder/neck/sleeve/inseam/outseam/thigh/height), applied uniformly to every garment type with no garment-specific set. **Verdict on the "is the generic set sufficient" question**: NOT sufficient for Blouse — real blouse tailoring needs front/back neck depth, armhole, and garment length, none of which the current 10 fields capture (bust is already covered by `chest`, waist by `waist`, shoulder by `shoulder`, sleeve by `sleeve`). Shirt is mostly covered (`neck` = collar size, `sleeve` = sleeve length) but is missing `cuff` (wrist measurement, distinct from sleeve length) and would benefit from `garmentLength`. Pant is already adequately covered by the existing set (waist/hips/inseam/outseam/thigh) — no new pant-specific field needed.
- **`languageLock`**: set identically in exactly 2 places — `industry-template.service.ts:254`'s `changeBusinessType()` and `setup.service.ts:53`'s `completeSetup()`, both as `isServiceTemplate ? 'en' : 'multi'`, no other reads/writes anywhere in the codebase. **Significant finding**: `languageLock` is currently **write-only** — nothing in the renderer reads it. The Language settings section (`SettingsScreen.tsx`'s `SECTIONS` array, `id: 'language'`) has no gate at all (`status: 'available'`, unconditional). Service-business screens appear English-only purely by convention (they simply never call `t()`), not because of any enforced runtime lock. This means: (a) the actual behavior change needed is wiring real `t()` calls into Tailoring's screens, not "unlocking" any UI gate that doesn't exist; (b) correcting the stored `languageLock` value is still worth doing for data correctness/future-proofing, in case something starts reading it later.
- **`TailoringScreen.tsx`**: single screen, ~680 lines, handles both Orders and Measurements tabs, zero `t()` calls (pure hardcoded English), confirmed matching the general service-business convention. Sidebar's Tailoring nav entry likewise has no `i18nKey` (`Sidebar.tsx:126`), consistent with the "Phase 22 — Service Business Foundation (English-only, no i18nKey)" comment covering all 24 service nav items.
- **Existing i18n pattern** (from `logistics` namespace, the most recent precedent): `useTranslation()` → `const { t } = useTranslation()` → `t('logistics.fleet.someKey')`; Sidebar items carry a `label` (English fallback) and optional `i18nKey`, rendered via `t(item.i18nKey, item.label)` (`Sidebar.tsx:230`) — i18next's built-in default-value fallback.
- **Serialization gotcha (must not regress)**: both `measurement-record.service.ts` and `tailoring-order.service.ts` have an explicit Decimal→Number serialization step (`serializeMeasurementRecord`/`serializeTailoringOrder`) because Electron's IPC structured-clone cannot carry a Prisma `Decimal` instance. Every new `Decimal` field added to `MeasurementRecord` must be added to `serializeMeasurementRecord` too, or every measurement-record IPC response will throw.
- No Zod validation layer exists for `measurementRecord`/`tailoringOrder` IPC handlers (payloads flow through as typed objects) — new fields need type additions in `channels.ts`/service function signatures, not a new Zod schema. `Product`'s create/update DOES go through Zod (`product.validation.ts`).

## 2. Scope delivered

1. **`Product.gender`** (`String?`, values `'MENS' | 'WOMENS' | 'UNISEX'`, nullable/optional) — added to schema, `product.validation.ts`'s Create/Update schemas, `product.service.ts`'s create/update data-building, and `ProductFormModal.tsx` (shown only when `isModuleEnabled('variant_tracking')`, matching the existing config-flag convention — CLOTHING/FOOTWEAR by default, any business that opts into `variant_tracking` gets it too, consistent with that flag's existing scope).
2. **`TailoringOrder.gender`** (`String?`, `'MENS' | 'WOMENS'`) and **`TailoringOrder.styleRegion`** (`String?`, `'INDIAN' | 'WESTERN'`) — added to schema, `tailoring-order.service.ts`'s create/update, `channels.ts`, and `TailoringScreen.tsx`'s order form.
3. **5 new `MeasurementRecord` fields** (`Decimal?`, inches, matching existing field style): `armhole`, `frontNeckDepth`, `backNeckDepth`, `garmentLength`, `cuff` — added to schema, `measurement-record.service.ts` (including the serialization function), `channels.ts`, and `TailoringScreen.tsx`'s `MEASUREMENT_FIELDS` array + measurement form.
4. **`languageLock` decoupling**: new exported `getLanguageLockFor(businessType: string): 'en' | 'multi'` helper in `industry-template.service.ts`, using a small `LANGUAGE_LOCK_EXCEPTIONS` Set (currently just `TAILOR_BOUTIQUE`) — called from both `changeBusinessType()` and `setup.service.ts`'s `completeSetup()` so the two existing call sites can never drift out of sync with each other again (matches this session's established pattern of single-source-of-truth fixes for exactly this class of duplication risk).
5. **Full i18n wiring for `TailoringScreen.tsx`**: new `tailoring` namespace in `en.json` (~60-70 keys covering every UI string on the screen — headers, labels, buttons, status labels, garment types, measurement field labels, placeholders, confirm dialogs), `useTranslation()` wired in, every hardcoded string replaced with `t('tailoring.xxx')`. Sidebar's Tailoring nav entry gets an `i18nKey: 'nav.tailoring'`. **Translated into all 12 non-English locale files** (ar/es/fr/gu/hi/id/kn/ml/mr/pt/ta/te) — real translations, not copy-pasted English (per the explicit lesson already logged in memory from Phase 45's discovery of exactly that anti-pattern elsewhere in the app).
6. **Agricultural Inputs note**: per the master plan, Agricultural Inputs needs zero special-case code for its own language support (it's a PRODUCT-category business type, already gets `languageLock: 'multi'` for free) — nothing to build here; that vertical is Phase 49's own scope.

## 3. Explicitly out of scope

- No new business types, no changes to any other service vertical's language lock.
- No changes to garment types themselves (`GARMENT_TYPES` list stays as-is) — this phase adds structure (gender/style/measurements) around the existing types, not new types.
- No retroactive backfill UI for existing `TailoringOrder`/`Product` rows to set the new optional fields — they default to `null`/unset, exactly like every other additive-nullable-column phase in this project.

## 4. Testing plan

- Unit: `measurement-record.service.ts` (new fields persist and serialize correctly, existing fields unaffected), `tailoring-order.service.ts` (gender/styleRegion persist on create/update), `industry-template.service.ts` (`getLanguageLockFor` returns `'multi'` for TAILOR_BOUTIQUE, `'en'` for every other service type, `'multi'` for every product type — and both `changeBusinessType`/`completeSetup` call sites actually use it).
- Live: launch the real dev app, switch to TAILOR_BOUTIQUE, confirm `languageLock` is persisted as `'multi'`, confirm the Tailoring screen's new gender/styleRegion fields and new measurement fields save and reload correctly through the real UI, confirm switching the app language actually changes TailoringScreen's rendered text (the real, previously-never-exercised path this phase unlocks).
