# Phase 50 — Healthcare Depth: Doctor Clinic Breadth Audit + New Vertical: Diagnostic & Pathology Labs: Completion Report

## Part 1 — Doctor Clinic Breadth Audit

**Audit question**: do the 5 existing clinic-family business types (`VET_CLINIC`/`GP_CLINIC`/`SPECIALIST_CLINIC`/`DENTAL_CLINIC`/`PHYSIO_CLINIC`) genuinely cover any doctor's clinic — `SPECIALIST_CLINIC` in particular, as a generic catch-all for any specialty (ENT, eye, gynecology, pediatrics, psychiatry, ayurveda/homeopathy, etc.) via its own configurable `ServiceCatalog`?

**Conclusion: the underlying data model genuinely holds.** `VisitNote`'s SOAP fields (subjective/objective/assessment/plan) are free text and specialty-agnostic; `ServiceCatalog` is fully generic (free-text `category`); `AppointmentsScreen.tsx` uses neutral "Provider" terminology throughout, not "Doctor"; no onboarding hint/tooltip infrastructure exists anywhere to narrow perception; no fixed enum of "consultation types" restricts what a specialist can name their services. Confirmed via direct reading of the schema and every clinic-adjacent screen, plus an independent Explore-agent sweep for hardcoded specialty/body-part/title assumptions.

**Two real, concrete gaps found and fixed (not hypothetical):**

1. **Hardcoded "Dr."/"Doctor Signature" text** in `VisitNoteScreen.tsx` (3 spots) and `ClinicalNotesListScreen.tsx` (1 spot) — every printed visit summary and clinical-notes list row forced a "Dr." title onto the provider's name regardless of their actual designation. This is not a hypothetical mismatch: `PHYSIO_CLINIC` shares this exact same screen (`visit_notes` module), and physiotherapists in India are conventionally **not** addressed as "Dr." (BPT/MPT, not a medical doctorate) — every physio clinic's printed visit summary was shipping a factually wrong title. Fixed by removing the hardcoded "Dr."/"Doctor" prefix everywhere, letting the provider's real name (and optional `specialization`) speak for itself, matching the "Provider" convention already used elsewhere in this same codebase.
2. **`SPECIALIST_CLINIC` was missing `token_queue`** (only `GP_CLINIC` had it) — walk-in token queues are just as common at specialist outpatient practices (ENT, eye, dermatology camps) as at GP clinics, a real coverage gap for "any specialty." Added `token_queue` to `SPECIALIST_CLINIC`'s `TEMPLATE_DEFAULTS`; the pre-existing `seedDefaultTemplates()` backfill mechanism (built in an earlier phase for exactly this scenario) propagates it to already-existing installs with no further code needed — verified with a new regression test.

**Deliberately not fixed, documented for a future phase**: `IndustrySettingsScreen.tsx`'s "Settings → Industry Template" screen only lists ~14 product-family business types — **none** of the 24 `SERVICE_TEMPLATE_TYPES` verticals (including the new `DIAGNOSTIC_LAB`) appear there at all. This means there is currently no in-app UI path to change a service business's sub-type or toggle its modules post-setup — only the one-time `SetupWizard` reaches them. This is a pre-existing gap (previously flagged in the Phase 42 and Phase 49 completion reports as out of scope) that this phase's own live-verification pass hit concretely a third time: verifying the new `DIAGNOSTIC_LAB` vertical required calling `window.api.industry.changeBusinessType()` directly rather than through any UI, since no picker tile exists. Still out of scope for this phase (a materially larger, separately-scoped UI-completeness project spanning all 24 verticals), but flagged here with harder evidence than before — worth a dedicated follow-up phase if the founder wants it prioritized.

## Part 2 — New Business Vertical: Diagnostic & Pathology Labs

**Design**: `DIAGNOSTIC_LAB` (SERVICE category, `languageLock: 'en'` — matches every other clinical vertical). Test/panel catalog deliberately reuses the existing `ServiceCatalog` model (a lab creates catalog rows like "Complete Blood Count"/"Lipid Profile"/"Chest X-Ray" with `category` = Pathology/Radiology/Panel) rather than inventing a parallel catalog, per the master prompt's explicit instruction.

**New schema** (additive migration `20260708120000_phase50_diagnostic_labs`, applied to the real dev DB via the project's established surgical `prisma migrate diff`/`db execute` technique — the CLI's schema-relative `DATABASE_URL` resolves to a different decoy file than the one the app actually opens, same gotcha already documented in project memory): `LabTestOrder` (one row per patient visit — order number, patient, optional customer/appointment/referring-provider links, status, sample-collection/reporting metadata, invoice link) and `LabTestOrderItem` (one row per ordered test — denormalized test name/category/price snapshot, per-item status, structured `resultParameters` JSON for pathology-style parameter+reference-range+flag results, free-text `resultSummary` for imaging/radiology-style findings).

**Workflow implemented** in `lab-test-order.service.ts`: create order with N tests → add/remove tests (only while `ORDERED`, before sample collection) → collect sample (bumps order + all items) → enter results per item (locks in `RESULT_READY`) → finalize report (requires every item `RESULT_READY`, cascades all to `REPORTED`) → mark delivered → generate invoice (reuses the exact atomic claim-sentinel pattern `appointment.service.ts`'s `generateAppointmentInvoice` already established, preventing a double-invoice race) → cancel/delete (both refuse an already-invoiced order).

**Permissions**: 3 new keys — `labOrders.view`/`labOrders.create` (Admin, Manager, **and Cashier** — front-desk registration/billing/report-handover is routine trust) and `labOrders.manage` (Admin, Manager only — sample collection, result entry, finalizing, cancel, delete; deliberately lab-technician-level trust, not Cashier-reachable).

**Frontend**: `LabOrdersScreen.tsx` (list + status tabs + create modal with a dynamic test-row builder that can pull from the `ServiceCatalog` or take an ad-hoc test name + detail modal with per-item inline result entry) and `LabReportPrint.tsx` (print component modeled directly on the existing `VisitSummaryPrint`, including `DocumentWatermark`/`AszurexMark` branding and the same clinical disclaimer convention). New route `/lab/orders`, sidebar nav entry gated on the new `lab_orders` module flag, `SetupWizard.tsx` tile.

## Independent review — 2 real, confirmed blocking issues found and fixed

A fresh-context review agent read every new/changed file in full and found:

1. **Finalized reports could be silently edited with no re-approval, permanently desyncing order/item status.** `updateTestResult`'s guard rejected `CANCELLED`/`DELIVERED`/`ORDERED` order statuses but not `REPORTED` — editing a finalized item's result flipped it back to `RESULT_READY` while the order stayed `REPORTED`, and `finalizeReport` unconditionally refuses an already-`REPORTED` order, so the order could never be re-finalized to reflect the correction. Concrete scenario: a manager corrects a Hemoglobin value from 13.5 to 9.5 after finalizing — the edit silently succeeds, but the printed report and the order's own status now permanently disagree with each other. **Fixed** by adding `REPORTED` to the rejected-status list — results are locked once finalized, matching how this project treats other finalized documents. New regression test added.
2. **Cashier was locked out of Print Report and Generate Invoice**, contradicting the design intent documented in `seed.ts`'s own comment ("Cashier can log a new order and see status/print an already-finalized report"). The entire action panel in `LabOrdersScreen.tsx` — including these two routine front-desk actions — was gated behind `canManage` (`labOrders.manage`, Manager/Admin only) instead of `canCreate`. Also inconsistent with this codebase's own precedent: `appointments:generateInvoice` gates on `billing.createInvoice` (which Cashier holds), but `labTestOrders:generateInvoice` gated on `labOrders.manage`. **Fixed** by splitting the action panel into a front-desk group (`canCreate`: Generate Invoice, Print Report) and a lab-technician group (`canManage`: Collect Sample, Finalize, Mark Delivered, Cancel, Delete), and changing the `labTestOrders:generateInvoice` IPC handler's permission check from `labOrders.manage` to `labOrders.create`.

Both fixes verified live (Section below), not just re-read.

## Deliberately not done (documented, not silently skipped)

- **`IndustrySettingsScreen.tsx`'s 24-service-vertical gap** (Part 1, above) — real, now hit three times across three phases, still out of scope for this one.
- **No new business type was added for any single specialty** (ENT-only, ophthalmology-only, etc.) — the audit's own conclusion is that the generic `SPECIALIST_CLINIC` + configurable `ServiceCatalog` genuinely covers this, so adding narrower types would be unrequested scope, contradicting the master prompt's explicit "don't default to 'add more clinic types' without proof the generic model can't represent a real specialty's workflow."
- **No radiology/imaging-specific structured fields** (e.g. a DICOM-adjacent image attachment) — `resultSummary`'s free text covers the "radiologist's report/impression" case, which is proportionate for a "Lite" product; a full imaging-management subsystem was never in scope.
- **Sample collection is tracked at the order level, not per-physical-tube** — one order can bundle several blood tests under a single sample-collection event, matching how most small labs actually work; splitting into a full sample-to-test many-to-many model was considered and rejected as disproportionate complexity for this product's scale.
