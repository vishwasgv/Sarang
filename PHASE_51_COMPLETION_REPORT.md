# Phase 51 — New Business Vertical: Blood Bank: Completion Report

## Scope delivered

Per `PRODUCT_HARDENING_MASTER_PROMPT.md`'s Phase 51 entry: a new PRODUCT-category business type, `BLOOD_BANK`, covering donor management, donation collection + screening, blood-group-segmented stock with expiry-critical tracking, and compatibility-aware issuance to recipients.

**Categorization**: PRODUCT category (not in `SERVICE_TEMPLATE_TYPES`), so `businessCategory: 'PRODUCT'` and `languageLock: 'multi'` fall out automatically. **Founder explicitly chose "English-only, skip translation entirely"** for this phase's 4 new screens (asked directly, since PRODUCT-category businesses normally get full 12-language support like `AGRI_INPUTS`) — a deliberate, approved scope decision, not a gap. The screens use plain hardcoded English strings, matching the pattern the SERVICE-vertical Lab Orders screens (Phase 50) already use.

**Audit finding that shaped the design**: the existing generic `batch_tracking`/`expiry_tracking` module's `BatchManagementScreen.tsx` hardcodes a 30-day "expiring soon" window. Blood components have materially different, much shorter shelf lives (platelets ~5 days, whole blood/packed RBC ~35–42 days, plasma/cryoprecipitate ~1 year frozen) — a fixed 30-day window would flag every platelet unit as "expiring" for its entire life, or under-flag urgency for others. Blood Bank therefore does **not** enable `batch_tracking`/`expiry_tracking` at all; it gets its own dedicated stock screen with a per-component-type `SHELF_LIFE_DAYS`/`EXPIRY_ALERT_DAYS` lookup table, while still reusing `ProductBatch` underneath as the stock ledger (same "reuse the data layer, build a bespoke view" pattern already used for Diagnostic Labs in Phase 50).

**New Prisma models**: `Donor` (medical eligibility fields — blood group, weight, last donation date, deferral status/reason/end-date — structurally distinct from `Customer`, which is a billing party, not a person with donation eligibility), `DonationCamp` (lightweight camp/event tracking), `DonationRecord` (links donor→camp→ProductBatch, tracks `screeningStatus` PENDING/PASSED/FAILED and `isIssued`), `BloodIssue`/`BloodIssueItem` (issuing specific units to a recipient; `BloodIssueItem.donationRecordId` is `@unique`, so a unit can only ever be issued once — a genuine DB-level guarantee, not just an application-level check). Every screened-PASSED donation becomes one `ProductBatch` row under a single shared "Blood Unit" placeholder `Product` (the third instance of the find-or-create-a-placeholder-product pattern already used twice in this codebase, for appointments and lab orders).

**Compatibility logic**: standard ABO/Rh matrix for whole blood/packed RBC; a genuinely **separate, reversed** matrix for plasma (AB is the universal plasma donor, O the universal plasma recipient — the opposite of the RBC rule); platelets/cryoprecipitate get no hard rule, advisory only. Exposed to the renderer via a `bloodBank:checkCompatibilityBatch` IPC channel so the Issue screen's pre-selection warning calls the real backend logic rather than re-deriving the matrix client-side — a flawed client-side approximation was caught and replaced with this real call during the build itself (documented below).

**Workflow** in `blood-bank.service.ts`: `createDonor` → `createDonationRecord` (rejects a currently-deferred donor) → `updateScreeningStatus` (PASSED creates the ProductBatch + updates `lastDonationDate`; FAILED never touches inventory; rejects re-screening) → `getBloodStock` (per-component-type expiry alerting) → `createBloodIssue` (decrements stock, records a compatibility note per item) → `cancelBloodIssue` (reverses stock, refuses an invoiced issue) → `generateBloodIssueInvoice` (atomic claim-sentinel pattern, third reuse of the same technique from `appointment.service.ts`/`lab-test-order.service.ts`).

**Permissions**: 3 keys mirroring the Phase 50 Labs precedent exactly — `bloodBank.view`/`bloodBank.create` (Admin, Manager, **and Cashier** — donor registration, donation intake, and invoice generation are routine front-desk trust) and `bloodBank.manage` (Admin, Manager only — screening decisions and issuing units, a compatibility-critical clinically-consequential action).

**Frontend**: 4 new screens under `src/renderer/src/modules/blood-bank/ui/` — `DonorsScreen.tsx`, `DonationsScreen.tsx`, `BloodStockScreen.tsx`, `BloodIssueScreen.tsx`. New routes, sidebar nav (gated on `requiredModule: 'blood_bank'`), and — unlike Phase 50's Diagnostic Labs — `BLOOD_BANK` **was** added to `IndustrySettingsScreen.tsx`'s TEMPLATES array (a peer of the other PRODUCT-family types already listed there), giving it a working post-setup picker tile that Diagnostic Labs still lacks.

## Self-caught issue during the build

While building the Issue screen's pre-selection compatibility preview, an initial client-side approximation flagged *every* non-platelet/cryo unit as "possibly incompatible" regardless of the recipient's actual blood group — duplicating (and getting wrong) logic that already lives correctly in the backend. Caught and fixed before the independent review: added a real `bloodBank:checkCompatibilityBatch` IPC channel and rewired the screen to call it (via a `useEffect` keyed on recipient group + selection), rather than re-deriving the ABO/Rh matrix client-side.

## Independent review — 2 real, confirmed findings, both fixed

1. **Permanent donor deferral was silently defeated.** `createDonationRecord`'s guard was `donor.isDeferred && donor.deferredUntil && donor.deferredUntil > new Date()` — for a donor marked `isDeferred: true` with no end date (the schema's own modeling of an indefinite/permanent deferral, e.g. a confirmed infectious-marker-reactive donor), the second condition was `null`/falsy and the whole check short-circuited to allow the donation anyway. **Fixed**: `donor.isDeferred && (!donor.deferredUntil || donor.deferredUntil > new Date())` — a missing end date now correctly means "blocked indefinitely." Two regression tests added (indefinite deferral rejected; a temporary deferral whose end date has passed is correctly allowed through).
2. **The deferral feature was completely unreachable from the shipped UI.** `updateDonor` (the only path that can ever set `isDeferred`/`deferralReason`/`deferredUntil`) was never called anywhere in the renderer, and `updateScreeningStatus`'s FAILED branch never auto-defers a donor either — meaning finding #1's guard could never actually fire in the real app. **Fixed** by adding a "Mark Deferred" / "Clear Deferral" action to `DonorsScreen.tsx`'s detail modal (reason + optional end-date, gated on `bloodBank.manage` since it's a clinically consequential decision) — deliberately a manual staff action, not an auto-defer-on-FAILED rule, since different failure reasons warrant very different deferral lengths and that judgment shouldn't be automated. The exact same guard-logic bug (missing-end-date-means-not-blocking) was found independently duplicated in the frontend's own `isEligibleNow()` eligibility-badge helper and fixed the same way.

## Live UAT caught 2 further real bugs neither review layer could have

1. **A genuine foreign-key violation on the very first real issuance**, invisible to every unit test because mocks don't enforce real FK constraints. `BloodIssue.issuedById` was schema'd as a relation to `Employee`, but the service populates it from the logged-in session's `userId` — and `User`/`Employee` are entirely separate ID spaces in this codebase with no correlation between them (confirmed by checking every other "who did this" audit field — `Invoice.createdBy`, `Expense.createdBy`, `Payment.recordedBy` — all correctly relate to `User`, not `Employee`, for exactly this reason). Every real issuance failed outright. **Fixed** by changing the relation to `User` (`@relation("BloodIssueIssuedBy")`), since the table was brand-new this phase and had shipped nowhere, the original migration file was corrected in place (not a follow-up migration) and the corresponding table on the real dev DB was rebuilt with the corrected FK via the project's standard surgical `prisma migrate diff`/`db execute` technique.
2. **A stale "Next Eligible: Now" placeholder immediately after marking a donor deferred**, found while re-verifying fix #1 live. `listDonors` computes a `nextEligibleDate` field (donor's last donation + 90 days) that `getDonor`/`updateDonor` never did — the new "Mark Deferred" UI refreshed its detail view from `updateDonor`'s raw (unenriched) response, silently losing the field and falling back to a wrong "Now." **Fixed** at the root: extracted a shared `computeNextEligibleDate()` helper used by both `listDonors` and `getDonor` (rather than a narrow client-side patch), and changed the frontend's post-update refresh to call `getDonor` instead of trusting `updateDonor`'s response directly. Two regression tests added confirming `getDonor` and `listDonors` compute the identical value for the same donor, and that it survives an update to an unrelated field.

Both live-caught bugs were verified fixed by re-running the exact same live scenario afterward — the FK fix by completing a real issuance end-to-end (compatibility note correctly recorded and displayed: *"AB+ recipient can receive WHOLE_BLOOD from O- donor — compatible."*), the placeholder fix by re-opening the same donor's detail view and confirming a real date renders both immediately after deferring and immediately after clearing the deferral.

## Live UAT — full pass

**Admin workflow**: switched to `BLOOD_BANK` via the real Settings → Industry Template picker tile → confirmed sidebar shows Donors/Donations & Screening/Blood Stock/Blood Issue → registered a donor (O-) → recorded a donation (Whole Blood) → screening PASSED (confirmed the unit entered stock) → Blood Stock screen showed the O- unit with a real expiry date → issued the unit to a recipient (AB+, compatible per the universal-donor rule, confirmed no false warning) → confirmed stock correctly dropped to 0 available afterward → marked the donor deferred (indefinite) → confirmed a subsequent donation attempt for that donor was correctly rejected with a visible error → restored business type to `MANUFACTURING`.

**Cashier permission-split verification** (a throwaway Cashier-role user): confirmed Cashier sees "New Donor"/"Record Donation" (has `.create`) and "Generate Invoice" on an issue, but does **not** see "Record Screening"/"Issue Units"/"Cancel Issue" (all `.manage`-only) — exactly matching the intended front-desk-vs-lab-technician trust split.

Dev DB left clean: all test donors/donations/batches/issues removed, throwaway Cashier user removed, admin password re-randomized and session cleared.

## Final state

0 TypeScript errors (both `tsconfig.node.json` and `tsconfig.web.json`), **754/754 tests** (was 722/722 at Phase 50's close — 32 new in `blood-bank.service.test.ts`, including the deferral-guard and nextEligibleDate-consistency regressions found during this phase's own review/live passes).

## Deliberately not done (documented, not silently skipped)

- **12-language translation** for the 4 new screens — founder's explicit call this phase, revisit as a follow-up if/when prioritized (unlike `AGRI_INPUTS`, which got full translation).
- **No donation-camp management UI** was built beyond the schema (`DonationCamp` exists and `DonationRecord` can link to one, but there's no dedicated camp CRUD screen this phase) — the master prompt scoped camp tracking as in-scope "if this phase's own audit finds it in scope"; the audit judged a lightweight linkable model sufficient for now, a full camp-management screen would be a reasonable, cheap follow-up if real usage demands it.
- **No auto-deferral on a FAILED screening result** — a deliberate scope call (see finding #2 above): different failure reasons warrant very different deferral lengths/permanence, and that's a clinical judgment call this "convenience tool" shouldn't make unilaterally. Staff can manually mark a donor deferred via the new UI action immediately after a FAILED result if they judge it warranted.
- **Platelets/cryoprecipitate get no hard compatibility rule** — matches real-world blood-bank practice (ABO-matching is preferred but not strictly required for these components in many protocols) rather than inventing an overly strict rule this tool can't fully validate; always shown as advisory only, with an explicit non-blocking, non-authoritative disclaimer in the Issue screen.
- **Sample/unit tracking stays at one-unit-per-donation granularity** (no splitting a single whole-blood collection into separately-tracked component sub-units) — proportionate for this product's scale; a full component-separation workflow (e.g. spinning whole blood into RBC + plasma + platelets as three trackable units) was considered and rejected as disproportionate complexity for a "Lite" tool.
